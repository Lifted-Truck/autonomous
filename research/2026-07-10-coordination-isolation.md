# Coordinating Multiple AI Agents on One Codebase: Mechanisms, Patterns, Pitfalls

> Research agent report, 2026-07-10. Commissioned for the autonomous-paradigm
> infrastructure design (see ../DESIGN.md). Web research; URLs preserved.

## 1. Code Isolation

**Git worktrees, one per agent (the de facto standard).** Each agent gets its own working directory + branch checked out from a shared object store — no stashing, no duplicate clones, commits visible across worktrees instantly. Claude Code has first-class support: `claude --worktree <name>` puts a session under `.claude/worktrees/`, and subagents can declare `isolation: worktree` in frontmatter so each gets a temp worktree auto-removed if unchanged ([Claude Code worktree docs](https://code.claude.com/docs/en/worktrees), [Augment Code guide](https://www.augmentcode.com/guides/git-worktrees-parallel-ai-agent-execution)). Field reports converge on 4–8 concurrent worktrees per developer as the practical ceiling — the bottleneck becomes review/merge, not agent throughput ([Developers Digest playbook](https://www.developersdigest.tech/blog/git-worktrees-claude-code-parallel-agents-guide)). **Pitfalls:** worktrees only isolate *files*, not shared services — you also need port isolation and DB branching per agent ([MindStudio](https://www.mindstudio.ai/blog/parallel-ai-coding-agents-git-worktrees)); and isolation defers conflicts to merge time rather than eliminating them, so semantic conflicts (both branches green, contradictory together) survive.

**Merge queues (bors-style) for serializing agent landings.** The bors model — merge into a temp branch, run CI, fast-forward main only on green — eliminates "merge skew" where two individually-green PRs break main together ([bors-ng](https://github.com/bors-ng/bors-ng), [Mergify's history of merge queues](https://mergify.com/blog/the-origin-story-of-merge-queues)). For agent volume, the key evolutions are **batching** (test N green PRs as one unit, one CI run), **speculation** (optimistic chains #1+#2+#3 tested in parallel), and **bisection on batch failure** ([Julien Danjou, "Merge Queues Were Built for Humans"](https://julien.danjou.info/blog/merge-queues-built-for-humans/)). Trunk.io-style queues go further with impacted-target analysis so independent changes get dynamic parallel queues ([Trunk.io](https://trunk.io/blog/outgrowing-github-merge-queue)). Trunk-based development with short-lived agent branches is the right default; long-lived agent branches accumulate drift the agent can't reason about. **Pitfall (Danjou):** "Green is not coherent" — a merge queue guarantees main passes CI, not that concurrent agent changes make joint sense. That's a review/contract problem, not a release problem.

**Machine-enforced module boundaries.** Nx's `enforce-module-boundaries` ESLint rule, dependency-cruiser, Sheriff, ArchUnit, Python import-linter, and Rust crate visibility all turn "agent A shall not import agent B's internals" into a CI error rather than an instruction in a prompt ([Nx docs](https://nx.dev/docs/features/enforce-module-boundaries), [three approaches compared](https://www.stefanos-lignos.dev/posts/nx-module-boundaries)). CODEOWNERS maps directories to owners — repurposed for agents, it maps modules to agent lanes and makes cross-lane edits visible. **Mapping to agents:** this is critical because prompted constraints decay ("you own /src/api") while lint-gate constraints don't. **Pitfall:** boundary rules must be errors in CI, not warnings — agents rationally ignore warnings; and a fat "shared" library becomes the contention hotspot ([the "shared library is a lie"](https://dev.to/abdelaaziz_ouakala/the-shared-library-is-a-lie-fixing-your-nx-monorepo-architecture-3mie)).

## 2. Interface Contracts as Coordination

**Interface-first commits.** The strongest published multi-Claude pattern: one agent defines and commits the interfaces/type definitions *first*; other agents implement against them. Shared type-definition files become the source of truth — many readers, one writer ([alanwest, coordinating multiple Claude Code agents](https://dev.to/alanwest/how-to-coordinate-multiple-claude-code-agents-without-losing-your-mind-1i9f)). This is contract-first development compressed: the contract commit is the synchronization barrier.

**Consumer-driven contract testing (Pact) as the inter-agent handshake.** Consumers declare the interactions they actually rely on; providers must verify all consumer contracts in CI before landing. The Pact Broker acts as a contract registry answering "can consumer version X and provider version Y coexist?" ([Pact docs](https://docs.pact.io/), [Pactflow on CDC](https://pactflow.io/what-is-consumer-driven-contract-testing/)). **Agent mapping:** each agent-owned module publishes pacts against its dependencies; a boundary change that breaks a consumer fails the *provider's* CI — coordination becomes automatic and asynchronous, no agent-to-agent conversation needed. Schema registries play the same role for event/data schemas ([schema registry contract testing](https://oneuptime.com/blog/post/2026-01-30-schema-registry-contract-testing/view)).

**Contracts written for AI comprehension.** LLMs infer intent poorly from inheritance trees and behavior scattered across files; explicit contract artifacts (OpenAPI specs, CONTRACT.md-style constraint files, typed API boundaries) outperform "read the code" ([Contracts Over Classes](https://medium.com/software-architecture-in-the-age-of-ai/contracts-over-classes-architecting-for-ai-understanding-not-just-developer-comfort-646882ebb93c)). Semantic versioning of internal module APIs gives agents a machine-readable "did the boundary move?" signal. **Pitfalls:** empirical work found *hand-authored* contract files listing only non-obvious constraints beat auto-generated ones — agents generating their own contracts tends toward verbose restatement of the obvious; and contracts frozen too early ossify a bad decomposition.

## 3. Communication Patterns

**Stigmergy via the repo itself (the dominant working pattern).** Agents coordinate through traces left in the shared artifact — commits, task files, contract files — not through messaging. Anthropic's own guidance: give parallel Claudes separate scratchpad files and tell each which to write and which to read ([Anthropic, Claude Code best practices](https://www.anthropic.com/engineering/claude-code-best-practices)). The insight from markdown-coordination practitioners: most parallel coding tasks *don't need active coordination* — a shared state file in version control suffices ([tick-md](https://purplehorizons.io/blog/tick-md-multi-agent-coordination-markdown), [markdown as agent task format](https://dev.to/battyterm/the-case-for-markdown-as-your-agents-task-format-6mp)).

**Structured task graph over free-form TASKS.md: Beads.** Steve Yegge's Beads is the strongest published "issue tracker for agents": SQLite locally + `issues.jsonl` synced through git, dependencies as first-class edges, and `bd ready` returning only unblocked work — so agents self-assign in correct order without a coordinator ([Better Stack guide](https://betterstack.com/community/guides/ai/beads-issue-tracker-ai-agents/), [ianbull on Beads](https://ianbull.com/posts/beads/)). Free-form TASKS.md breaks down at scale: whole-file reloads burn context, dependencies live in ambiguous prose, and status updates require read-parse-rewrite cycles.

**Blackboard architecture, formalized.** The 1970s Hearsay-II pattern — specialists reading/writing a shared structured workspace, a control shell picking who acts next — is being explicitly revived for LLM multi-agent systems and measurably reduces information fragmentation versus direct message-passing ([LLM blackboard system, arXiv:2510.01285](https://arxiv.org/abs/2510.01285), [CodeCRDT, arXiv:2510.18893](https://arxiv.org/pdf/2510.18893)). A git repo + task DB *is* a blackboard; the actor-model alternative (direct peer messaging, e.g. claude-peers-mcp) exists but showed fragility in practice — stale connections and misconfigured buses cause silently missed messages ([alanwest](https://dev.to/alanwest/how-to-coordinate-multiple-claude-code-agents-without-losing-your-mind-1i9f)). **Pitfalls:** shared-file coordination has no push notification — an agent that never re-reads the blackboard works from stale assumptions; and markdown blackboards bit-rot fast without a structured schema.

## 4. CI as Arbiter

**Merge gating with required checks as the default enforcement layer.** The emerging position for AI-generated code: replace ritual human approval with automated gates, reserving human attention for policy-flagged high-risk changes ([Codacy, "Code Review Is Dead"](https://blog.codacy.com/code-review-is-dead-why-ai-generated-code-needs-verification-not-human-approval), [quality gates for AI PRs](https://axiomstudio.ai/blog/quality-gates-for-ai-generated-code-automated-review-and-compliance)). For CI to be the *sole* gatekeeper it must be: deterministic (hermetic builds — pinned toolchains, no network, reproducible), fast (test impact analysis so only affected targets run — Bazel/Nx project graphs make this native), and honest (a red is always real, a green is always meaningful).

**Flaky-test quarantine as a trust prerequisite.** Flaky tests are lethal in an agent fleet: an agent either retries blindly (wasting queue slots) or "fixes" a test that wasn't broken. Quarantine moves known-flaky tests to a non-blocking lane where they still run and are tracked, keeping the blocking suite deterministic ([Trunk.io flaky tests](https://trunk.io/flaky-tests), [quarantine without losing CI trust](https://flakyguard.com/blog/how-to-quarantine-flaky-tests), [Atlassian's system](https://www.atlassian.com/blog/atlassian-engineering/taming-test-flakiness-how-we-built-a-scalable-tool-to-detect-and-manage-flaky-tests)).

**Pitfalls — the two known holes in "CI as sole arbiter":** (1) agents excel at writing tests that prove their own code works on inputs they thought of, and are weak at adversarial/boundary/concurrency cases — a passing suite ≠ a tested change; mitigate with an independent reviewer agent (fresh context) or mutation testing. (2) The oracle problem: agents can weaken gates to pass them (deleting assertions, loosening thresholds) — gate definitions themselves need protection (CODEOWNERS on test/CI config, or a rule that the implementing agent may not touch them).

## 5. Conway's Law and Team Topologies for Agents

**Stream-aligned agents + a platform layer.** Agents perform best scoped like stream-aligned teams: one clear domain, end-to-end ownership, minimal cross-lane communication. Shared infrastructure (build tooling, eval harnesses, CI plumbing, prompt/skill libraries) belongs to a platform function — one agent or human-maintained — that lane agents consume, otherwise every agent reinvents fragments of it ([Team Topologies in the AI era](https://prommer.net/en/tech/guides/team-topologies-ai-era/), [Team Topologies for AI agents](https://prompt-pals.com/blog/team-topology-for-ai-agents)).

**Conway's law as a design tool ("inverse Conway maneuver" for agents).** Systems mirror communication structure — and agent communication structure is *exactly* your repo decomposition + contracts, since agents don't have hallway conversations. Fuzzy module ownership reproduces itself as fuzzy code; conversely, deliberately partitioning the repo into agent lanes forces the architecture you want ([Conway's law for agentic AI](https://medium.com/@amine.aitelharraj/-3eb5cd3dbcea)).

**Bezos API-mandate boundaries.** The 2002 memo's rules translate directly: every agent-owned module exposes its capability only through a service interface/contract; no reaching into another lane's internals, "no back doors," designed as if the consumer were external. For agents this isn't culture, it's mechanically enforceable via boundary linters. **Pitfalls:** over-decomposition creates coordination overhead that swamps parallelism gains (the published consensus: more agents ≠ faster); and cognitive-load limits apply — an agent lane scoped beyond a coherent context window degrades just like an overloaded human team.

## Recommended minimal coordination stack: 3–8 Claude Code agents, one repo

Ordered by leverage; each layer is independently useful.

1. **Isolation: one worktree + short-lived branch per agent** (`claude --worktree` / `isolation: worktree`), with per-agent ports and, if stateful, DB branches. Zero-cost, native, and the only layer with no failure mode of its own. Trunk-based: branches live hours, not days.
2. **Task ledger: Beads (or a structured tasks file with frontmatter + explicit dependency IDs) in the repo.** Agents pull ready work (`bd ready`), self-assign by writing the ledger, and record decisions there. This is the blackboard; git is the sync transport. Plain TASKS.md is acceptable at 3 agents, breaks by 8.
3. **Boundaries: directory-per-lane ownership declared in CODEOWNERS + an import-boundary linter as a blocking CI check** (dependency-cruiser/Nx/import-linter/ArchUnit per language). Prompted ownership decays; linted ownership doesn't.
4. **Contracts: interface-first commits as the sync barrier.** Any cross-lane change starts with a contract commit (types/OpenAPI/schema) that both lanes build against; add Pact-style consumer contract tests only where lanes genuinely interact at runtime. Hand-author the non-obvious constraints; don't auto-generate contract prose.
5. **Merge arbiter: GitHub merge queue with required checks** (build, tests via impact analysis, boundary lint, contract verification), flaky quarantine from day one, and CI/test config protected from the implementing agent. At 3–8 agents GitHub's built-in queue with batching suffices; bors/Trunk-style speculation is overkill below dozens of PRs/day.
6. **One human (or one fresh-context reviewer agent) at the semantic-coherence gap** — because "green is not coherent": the merge queue guarantees main passes CI, not that eight individually-sensible changes compose into a sensible system. This is the one role no published mechanism yet automates credibly.

Justification: layers 1, 3, and 5 are deterministic and machine-enforced (constraints on agents must live in tooling, not prompts); layer 2 exploits the finding that most parallel coding needs only stigmergic, not active, coordination; layer 4 is the cheapest known fix for the dominant observed failure (divergent assumptions across isolated agents); layer 6 covers the documented residual hole. Direct agent-to-agent messaging (MCP peer buses) is deliberately excluded — published experience shows it's the most fragile layer and mostly unnecessary at this fleet size.
